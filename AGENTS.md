# AGENTS.md

## 项目
构建一个面向小微企业的游艇售卖网站。

该网站用于展示游艇产品、承接客户询盘、提供企业介绍与联系方式，并支持后台进行游艇信息和客户询盘管理。

## 当前阶段
项目当前处于：

**项目初始化与基础骨架搭建阶段**
向
**可演示、可管理、可上线的游艇销售 MVP**
推进阶段。

当前不是大型电商平台，也不是多商家入驻平台，更不是复杂 CRM / ERP 系统。

## MVP 范围
当前优先保证以下能力：

1. 前台首页展示
2. 游艇列表页
3. 游艇详情页
4. 关于我们页
5. 联系我们页
6. 客户询盘提交
7. 管理员登录
8. 后台游艇管理
9. 后台询盘查看
10. 基础图片展示与图片字段管理
11. 最小可用的演示闭环

## 技术栈
- Frontend: Next.js + TypeScript + Tailwind CSS
- Backend: Next.js Route Handlers
- Database: PostgreSQL
- ORM: Prisma
- Auth: 基础管理员鉴权
- Image strategy: 优先支持图片 URL 字段，后续可接入 Cloudinary
- Deployment: Vercel 或 Ubuntu

## 目录方向
推荐向以下结构收口：

- `src/app`: 页面与路由
- `src/components`: 公共组件
- `src/lib`: 工具、数据库、鉴权封装
- `src/server`: 服务端业务逻辑（若后续需要）
- `prisma`: 数据库 schema 与迁移
- `public`: 静态资源

路由方向应至少覆盖：

- `/`
- `/yachts`
- `/yachts/[id]`
- `/about`
- `/contact`
- `/admin/login`
- `/admin/yachts`
- `/admin/inquiries`
- `/api/...`

## 核心方向
1. 保持主业务链路稳定：
   - 首页展示
   - 游艇列表
   - 游艇详情
   - 询盘提交
   - 后台游艇管理
   - 后台询盘查看

2. 当前重点：
   - 项目骨架
   - 数据模型
   - 基础页面路由
   - 询盘闭环
   - 后台最小管理闭环
   - 可运行、可演示、可继续迭代的工程基础

3. 当前不做：
   - 在线支付
   - 多商家入驻
   - 多语言系统
   - 实时聊天系统
   - 复杂权限系统
   - 高级 CRM
   - AI 推荐系统
   - 复杂订单与合同流

4. 优先做：
   - 可演示
   - 可维护
   - 可验证
   - 可扩展
   - 可上线

## 工作规则
1. 每次开始前先阅读本文件。
2. 优先遵守 MVP 范围，不随意扩功能。
3. 优先保证真实业务闭环，不优先堆页面数量。
4. 优先做小步、可验证迭代。
5. 能复用已有组件、工具、页面结构就不要重造。
6. 不要虚构不存在的数据库字段或后端能力。
7. 不要为了“看起来完整”而先做大量假数据页面。
8. 进入新一轮开发前，先判断当前任务属于：
   - 后端能力
   - 前端实现
   - 前端设计
   - 部署与演示
   然后只推进一个最小目标，不要同一轮无节制扩散。

## 开发顺序规则
默认按以下顺序推进，除非用户明确调整优先级：

1. 项目初始化
2. 数据库 schema
3. 前台列表 / 详情主链路
4. 询盘提交闭环
5. 后台登录
6. 后台游艇 CRUD
7. 后台询盘查看
8. 部署准备
9. UI 收口

## 前端分工规则
1. 前端**设计**任务优先由 Antigravity 完成。
2. 前端设计包括：
   - 页面信息层级
   - 页面布局结构
   - 视觉风格收口
   - 页面交互流程
   - 演示展示方式设计
3. Codex 负责前端**实现与验证**，包括：
   - 页面编码
   - 状态处理
   - 表单逻辑
   - API 对接
   - aria 语义
   - build / typecheck / smoke 修复
4. 如果任务同时包含设计和实现，应优先由 Antigravity 完成设计，再由 Codex 实现。
5. 不要把 Codex 和 Antigravity 的指令混在同一轮里下发。
6. 默认先由 Codex 连续推进后端和工程基础，达到较完整阶段后，再切换到 Antigravity 做前端设计收口。

## 前端设计与 UI/UX 规则
1. 只要任务包含前端设计、页面收口、交互优化、信息层级整理，默认应使用 **UI/UX Pro Max**。
2. 使用 UI/UX Pro Max 时，必须落到具体页面点上，例如：
   - 首页首屏信息层级
   - 列表页筛选与卡片层级
   - 详情页参数区与询盘区主次关系
   - 后台列表页操作按钮可见性
   - 成功 / 失败 / 空状态表达
3. 若本轮前端任务明确涉及设计，输出中应说明：
   - 是否使用了 UI/UX Pro Max
   - 具体落在什么页面点
   - 为什么这样收口
4. 前端设计收口优先：
   - 信息清楚
   - 主操作明显
   - 状态明确
   - 路径闭环
   - 演示友好
5. 不要为了炫技而无节制重做页面；优先做最小高价值收口。

## 跨平台兼容规则
1. 项目开发与脚本实现默认需要同时兼容：
   - Ubuntu 24
   - Windows
2. 编写脚本、命令、路径处理、环境变量注入时，优先采用跨平台写法。
3. 非必要不要写死：
   - `cmd.exe` 专属行为
   - PowerShell 专属行为
   - bash-only 语法
   - Linux/Windows 单边路径分隔符
4. 涉及路径拼接时，优先使用语言内建跨平台能力。
5. 涉及脚本或启动命令时，应明确：
   - 哪一侧已验证
   - 哪一侧未验证
   - 未验证的剩余风险

## 仓库治理规则
1. 新增文件前，先检查同目录和相邻目录是否已有职责相近文件；能复用就不要新建平行文件。
2. 如果确实需要新增同类文件，必须说明：
   - 为什么不能复用旧文件
   - 两者职责边界
   - 哪一份是后续主入口
3. 临时文件、实验文件、复制文件、备份文件不要长期留在仓库中。
4. 不允许留下 `tmp_*`、`test_*_copy`、`backup_*`、`draft_*` 这类无正式职责命名文件作为长期残留。
5. 正式页面、正式脚本、正式文档如果已进入主路径，应纳入 Git 跟踪。

## Bug 修复规则
1. 修 bug 优先做根因修复，不要只做表面补丁。
2. 不要用硬编码、吞错、跳过真实分支的方式伪装修复完成。
3. 如果必须临时兼容，必须说明：
   - 根因
   - 为什么当前只能临时处理
   - 剩余风险
4. 如果同一问题影响多个页面或模块，优先抽公共修复点。

## 文档与注释规则
1. 所有新写或修改的 Markdown 内容默认使用中文。
2. 所有新写或修改的代码注释默认使用中文。
3. 外部标准术语可保留英文，但整句说明优先中文。
4. 以下技术契约值保持原样：
   - API 路径
   - 表名
   - 字段名
   - 类型名
   - 脚本名
   - 机器可读返回值
5. 如果文档已经成为执行路径的一部分，更新时优先保证与真实代码、真实脚本一致。

## 验证规则
1. 每次有实际改动都应尽量附带验证。
2. 优先使用：
   - `typecheck`
   - `build`
   - `test`
   - 定向 smoke
3. 不要只用“代码看起来没问题”作为完成依据。
4. 除非任务仅涉及纯文档，否则不要在没有至少一种可执行验证的情况下声称完成。
5. 如果当前环境无法完成运行验证，必须说明：
   - 本应验证哪一步
   - 为什么当前不能验证
   - 剩余风险是什么

## 运行验证优先级
1. 前端改动：优先验证页面能打开、交互能执行、关键状态可见。
2. 后端改动：优先验证接口能请求、关键返回符合预期。
3. 表单改动：优先验证提交成功、失败反馈、字段校验与空状态。
4. 后台管理改动：优先验证新增、编辑、删除、列表回显闭环。
5. 跨平台相关改动：优先验证至少一侧真实执行，并明确另一侧风险状态。
